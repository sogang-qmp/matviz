## What landed

- Removed `CrystalDocument` class and `openCustomDocument` method.
- `CrystalEditorProvider` now implements `vscode.CustomTextEditorProvider`
  instead of `CustomReadonlyEditorProvider<CrystalDocument>`. The
  webview and the text buffer share the same `vscode.TextDocument` —
  VSCode's standard dirty state, save dialog, and external-change
  detection flow through automatically.
- New private helper `parseDocument(document: TextDocument):
  ParsedContent | null` replaces the old `openCustomDocument` parse
  path. Returns `null` on parse failure (caller decides toast vs.
  silent drop).
- New private interface `ParsedContent` carries `{ uri, trajectory,
  volumetric?, structure }` — same payload the rest of the provider
  consumes (export, info display).
- Initial parse on editor open: failure shows a "(Open as Text)" toast
  but the webview still mounts (vs. throwing and refusing to open as
  the previous behavior did).
- **Debounced live-edit reparse** via
  `vscode.workspace.onDidChangeTextDocument`:
  - 250 ms debounce after the last keystroke.
  - `parseSeq` token discards stale parse results when the user types
    fast (concurrent reparses from rapid edits).
  - Edits filter to the URI of the matviz-viewed document only; other
    open docs don't trigger reparse.
  - Parse failure during edit silently keeps the last good state — no
    flickering, no broken view. Future v0.20.x could surface a
    "(stale: parse error)" badge in the info pill.
- `onDidDispose` now also disposes the change-event subscription and
  clears any pending debounce timer (no leaks).

## New command (split-pane workflow)

- `matviz.openWithText` — opens the text editor for the currently-
  active matviz tab in `ViewColumn.Beside`. Surfaces the split-pane
  affordance now that live edit is wired.
- Registered in `package.json` `contributes.commands` with title
  "MatViz: Open Text Editor (split pane)".
- Wired in `src/extension.ts` mirroring the existing `matviz.openInViewer`
  command pattern.

## Behavior preserved

All existing commands and message handlers route through unchanged:
- `matviz.resetCamera`, `matviz.toggleBonds`,
  `matviz.viewAlongDirection`, `matviz.viewNormalToPlane`,
  `matviz.addLatticePlane`, `matviz.clearLatticePlanes`,
  `matviz.screenshot`, `matviz.exportCif`, `matviz.exportPoscar`,
  `matviz.showWulff`, `matviz.clearWulff`, `matviz.addPhase`,
  `matviz.clearPhases`, `matviz.compareToPhase`, `matviz.clearComparison`
  — all still work via `provider.postMessageToActive(...)` which is
  unchanged.
- Webview message handlers unchanged: `ready`, `openAsText`,
  `addPhaseRequest`, `comparisonResult`.
- `exportStructure(format)` unchanged surface, internally reads from
  `activeContent.structure` instead of `activeDocument.structure`.

## Observable behavior changes

1. **Live edit reflows the 3D view** — the headline feature.
2. **Save semantics**: `Ctrl+S` works on the text-pane companion
   (saves the underlying file). Previously the readonly provider
   ignored saves.
3. **Dirty marker**: VSCode's tab-dirty dot now appears when the text
   buffer has unsaved edits. Was always-clean before (read-only).
4. **External change pickup**: `git pull`, "Revert File", or external
   editor saves now reflow the view automatically. Previously the
   matviz editor cached the initial parse and never re-read.
5. **Parse failure UX**: on initial open, the matviz tab now opens
   *anyway* with an empty webview + a toast. Previously it threw and
   refused to open at all (fell through to the default text editor).
