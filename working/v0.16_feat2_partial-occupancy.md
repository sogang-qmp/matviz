## Autonomous judgments recorded

### J1. Per-atom Mesh vs InstancedMesh-with-bucketing

대안:
- A. InstancedMesh per `(element, occupancy_bin)` (예: 0.05 step → 20 bins).
  같은 (element, occupancy) 사이트는 한 mesh에 묶임. 효율적이지만 quantization
  loss.
- B. **Per-atom Mesh ✅ 선택.** 정확한 per-site opacity. 부분 점유 사이트는 보통
  드물어 (≤ 50 in typical CIF) 성능 영향 미미.

근거: plan §16.2의 통과 기준 "혼합 점유 사이트가 점유 비율로 시각화" — 정확한
per-site 비율이 더 중요.

Material 캐시는 같은 rebuild 내 같은 `(color, opacity)` 쌍 재사용 → 메모리
효율 적당히 유지.

### J2. Aniso 사이트가 occupancy<1일 때 우선순위

플랜 명시 없음. 자율 결정:
- Ellipsoid path 우선 → 부분 점유 사이트인 동시에 aniso인 atom은 ellipsoid로
  렌더 (opaque). partial로 라우트되지 않음.
- 이유: ellipsoid가 더 정보가 풍부한 표현. transparent ellipsoid (occupancy
  반영) 처리는 16.x로 deferred.

코드 위치: `renderer.ts buildAtoms()` partial 분기에서 `if (ellipsoidIdxSet.has(i)) continue;`.

### J3. depthWrite: false + renderOrder: 1

부분 점유 사이트는 같은 좌표에 stacked될 가능성 (Mg+Fe at origin). depthWrite
true면 첫 atom이 zbuffer를 채워 두 번째 atom occlude. depthWrite: false로
blending 가능.

부작용: opaque atoms와의 ordering이 정확하지 않음 (renderOrder=1로 후순위 처리
하지만 stacked partial atoms 끼리의 정렬은 정의 안 됨). 일반적인 mixed 사이트는
2-3종이라 시각적으로 무난.

### J4. 영속화 정책 — 16.1 ellipsoid와 일관

`showPartialOccupancy`도 영속화 (16.1 ellipsoid처럼). `showPolyhedra`만 매 파일
오픈마다 off로 리셋되는 special case. 부분 점유 데이터가 있는 파일을 주로 다루는
사용자는 매번 다시 켜는 것 비효율.

비-occupancy 파일 로드 시 섹션 자동 숨김, 토글 state는 유지.

### J5. Test fixture 합성

calcite/anorthite 같은 실제 광물 fixture는 라이선스/소싱 비용 있음. 합성 P1
cubic + Mg/Fe mix가 parser contract 검증에 충분. 실제 광물 검증은 user-side.

### J6. 시각 검증 한계

CLI 렌더러에 부분 점유 path 없음 (16.0 결정 carry-over). visual harness는 회귀
catching 한정 (4 fixtures 모두 occupancy 데이터 없으므로 ΔRGB=0).

실제 transparent 렌더링 시각 검증은 user-side VSCode install + test-occupancy.cif
열기 → 토글 on → Mg/Fe 두 atom이 stacked transparent로 표시.

## Known limitations / deferred

1. **Aniso + occupancy 동시 사이트**: ellipsoid path 우선, opacity 미반영.
   v0.16.x.
2. **Partial atoms picking 미검증**: per-atom Mesh이므로 raycaster가 동작
   가능하지만 atomMeshMap에 등록 안 함 — 클릭 시 atom info 토스트 안 나옴.
   v0.16.x.
3. **POSCAR/CIF 외 파서 occupancy 미지원**: POSCAR는 부분 점유 표현 없음
   (format 한계). XYZ/PDB/Cube 동일.
4. **Stacked atoms ordering 미정의**: 같은 사이트 partial atoms 끼리 정렬.
   2-3종 mix는 시각적으로 무난, 더 많으면 artifacts 가능.
5. **Bond detection은 occupancy 무시**: 부분 점유 사이트 사이의 bond는 가장 큰
   점유 종 기준. 현재 거동 유지.
