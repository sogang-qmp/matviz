## Autonomous judgments recorded

### J1. CrystalDocument 백워드 호환 — `.structure` getter 유지

CrystalDocument의 기존 사용 사이트는 `.structure` 필드 직접 접근 (export 함수,
info display 등). 이를 trajectory로만 바꾸면 ~5개 호출 사이트 모두 `.frames[0]`
로 수정 필요.

**선택:** `trajectory`를 primary 필드로 추가, `structure: CrystalStructure`를
**getter**로 유지 (`get structure() { return this.trajectory.frames[0]; }`).
Single-frame 사용 사이트 zero-touch.

### J2. setFrame이 loadStructure보다 lightweight

loadStructure는 bondParams clear, selection clear, measurements clear,
axisIndicator update, full rebuild을 수행. 재생 중 매 frame마다 이걸 다 하면
사용자 state 손실 + perf 저하.

**선택:** setFrame은 lightweight:
- bondParams 보존 (사용자 설정).
- selection 보존 (atom-index 안정 across frames).
- measurements 보존.
- axisIndicator는 `latticeMode='per-frame'`일 때만 업데이트.
- rebuild만 호출.

loadTrajectory는 첫 진입에 full reset (loadStructure와 같음).

### J3. crystalEditorProvider도 trajectory entry로 즉시 전환

대안:
- A. crystalEditorProvider는 기존 `parseStructureFile + loadStructure` 유지,
  17.1.1+에서 multi-frame 진입 시 추가.
- B. **17.1.0에서 `parseStructureFileTraj + dispatch` 즉시 도입**, 1-frame은
  loadStructure로 라우트.

선택 B 근거:
- dispatch 코드 path가 lock-in되어 17.1.1에서 AXSF가 multi-frame 반환하면
  자동으로 loadTrajectory 진입.
- 1-frame 케이스의 dispatch 비용은 한 번 if 검사 — perf 무시 가능.
- 17.1.0 commit이 "infra ready" 상태로 closure — 17.1.1이 단순 parser 추가만
  하면 됨.

### J4. main.ts message handler에서 helper factor 회피

원래 계획: `postLoadSetup()` helper로 loadStructure + loadTrajectory 케이스
공유. 그러나 Edit tool이 `Å³` literal escape sequence 매칭에
실패 (rendered `Å³` vs source escape) — 안전 우회로 코드 inline 중복 (9 LOC).

대안 (helper factor + 중복 제거)는 향후 17.1.4에서 trajectory section 추가
시 자연스럽게 같이 정리.

### J5. v0.16 옵셔널 필드 (thermalAniso/occupancy/magMom) 보존

테스트 5에서 검증: `parseStructureFileTraj`가 파서 단계에서 채워진 옵셔널
필드를 `frames[0]`에 그대로 보존. v0.16 기능 + v0.17 trajectory mode가 같이
동작 (v0.16 테스트 fixture가 trajectory wrap 후에도 magMom 필드 가짐).

`frames[i].thermalAniso` 등 per-frame 변동은 17.1.3 extended XYZ properties
파이프라인에서 가능 — 17.1.0 첫 컷에서는 frame[0]의 옵셔널만 의미.
