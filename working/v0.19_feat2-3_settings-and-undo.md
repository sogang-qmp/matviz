## 19.2 — `matviz.defaults.*` settings namespace

### Schema (`package.json` `contributes.configuration`)

7 settings, all under `matviz.defaults.*`:

| Key | Type | Default | Description |
|---|---|---|---|
| `style` | enum | `ball-and-stick` | Default display style |
| `palette` | enum (dark/light) | `dark` | Default CPK palette |
| `showBonds` | boolean | `true` | |
| `showBoundary` | boolean | `true` | Periodic-image boundary atoms |
| `cameraMode` | enum (orthographic/perspective) | `orthographic` | |
| `bondCutoffPad` | number (0..2) | `0.3` | Padding (Å) added to r_A + r_B for bond cutoff |
| `axisIndicatorSize` | number (40..400) | `80` | |

All settings render in VSCode's settings UI under the "MatViz" section.

### Wiring

- `crystalEditorProvider.ts:resolveCustomTextEditor` reads
  `vscode.workspace.getConfiguration('matviz.defaults')` and embeds the
  resolved values as a `window.MATVIZ_DEFAULTS` JSON literal into the
  webview HTML.
- `src/webview/main.ts` reads `window.MATVIZ_DEFAULTS` immediately after
  the renderer is constructed but BEFORE `vscode.getState()` restoreState
  runs. So the precedence is: settings → renderer constructor defaults
  (which the settings overwrite) → localStorage (which overwrites both
  on return visits to a previously-opened URI).
- New setters added to `CrystalRenderer` for parity:
  `setShowBonds(v)`, `setShowBoundaryAtoms(v)`. The existing toggle*
  methods stay; the new `set*(v)` methods are used by the defaults
  apply path.

### Settings vs localStorage interaction

Documented inline in main.ts and in the setting descriptions:
> Settings provide initial defaults for fresh opens. localStorage state
> wins on return visits to the same URI.

This matches the user's expectation that "I changed the setting, why
isn't this saved file using it?" → "open it fresh (clear webview cache)
or accept that prior personalisations override defaults".

## 19.3 — Property-panel undo/redo

### Stack design (`src/webview/undoStack.ts`, 50-line module)

- `UndoStack` class: `record(undo, redo)` / `undo()` / `redo()` /
  `canUndo()` / `canRedo()` / `clear()`.
- Cap: 50 entries. Recording a new action clears the redo stack
  (linear history, no branching).
- Session-scoped — not persisted via vscode.setState. Matches VSCode
  text-editor undo semantics.

### Tracked operations

In `buildAtomPropsUI`:
- **Element color**: `colorInput.change` records before/after. Live
  `input` events still drive immediate visual feedback during drag —
  the undo entry only commits when the picker closes.
- **Element radius**: same dual-event pattern (`input` for live preview,
  `change` for undo commit).
- **Element visibility**: single `change` event, single undo entry per
  toggle.

In `buildBondPropsUI`:
- **Per-pair bond enable** (`vis` checkbox): single `change` event.
- **Per-pair bond min/max**: combined `commitBondCutoff` records both
  values together — undoing a min change also restores max if it was
  edited in the same commit (and vice versa). Tracking value changes
  via per-row `bondMinBefore`/`bondMaxBefore` closures.

### Keyboard binding

Added to the existing `window.keydown` handler in main.ts. Skips when
focus is in `<input>`/`<textarea>` (the early-return at line 1041
already covers this) so the side-by-side text editor's own undo still
works as expected when text-pane has focus.

```
Ctrl+Z  /  Cmd+Z       → undoStack.undo()
Ctrl+Shift+Z / Cmd+Shift+Z → undoStack.redo()
```

### Stack invalidation on file load

Both `loadStructure` and `loadTrajectory` message handlers now call
`undoStack.clear()` before loading the new structure. Without this,
undoing a color change after switching files would try to recolor an
element that may not exist in the new structure (no-op, but
semantically nonsensical).
