# v0.18.0 post-release polish (2026-04-27)

User feedback after the v0.18.0 build, gathered in one polish iteration:

1. **Phases → Overlay** — section renamed, collapsed by default
2. **Step inputs narrower** (toolbar Step(°) / Step(%))
3. **Supercell stepper narrower** (per-axis cells fixed at 64px)
4. **Atoms + Bonds list visibility checkboxes → iOS switches** (small variant)
5. **Bond min/max inputs unified** with the toolbar `.num-wrap` style
6. **Top toolbar responsive** — wraps onto multiple rows + shrinks at narrow
   widths instead of horizontally scrolling

## Changes by file

### `src/editor/crystalEditorProvider.ts`
- `phases-section` panel-section now uses `panel-label-toggle` pattern
  (matches Atoms / Bonds / Polyhedra centers / Options). Label text changed
  from "Phases (overlay)" → "Overlay". All section content wrapped in
  `<div id="phases-content" class="hidden">` so the section starts
  collapsed.
- "+ Add Phase…" → "+ Add Overlay…", "Compare to first phase" →
  "Compare to first overlay" — wording aligned with the new section name.

### `src/webview/main.ts`
- New `initTogglePanel(phasesToggleBtn, phasesContent, 'Overlay')` call
  alongside the existing 4 toggleable sections.
- `setupNumberStepper` is now **float-aware**: reads `data-step` and
  `data-precision` from the wrapper, uses `parseFloat` and step-multiplier
  apply, blur-formats with the configured precision. Backward-compatible —
  default `data-step="1"` and `data-precision="0"` mirror the original
  integer behaviour for step-angle / step-zoom / sc-a/b/c.
- New `injectSwitch(input, sm)` helper that wraps the existing logic plus
  an explicit click-forward for bare (non-`<label>`-wrapped) inputs.
  `setupToggleSwitches()` now also walks `input.prop-vis[type=checkbox]`
  and injects the `.switch.sm` (30×18) variant.
- `buildAtomPropsUI`, `buildBondPropsUI`, `buildPolyCentersUI` each call
  `setupToggleSwitches(propsContainer)` after building rows so newly
  inserted `.prop-vis` checkboxes pick up switches.
- `buildBondPropsUI` now creates `.num-wrap.bond-num` wrappers with
  `data-step="0.05"` / `data-precision="2"` for min/max instead of native
  `<input type=number>`. The float-aware `setupNumberStepper` handles
  Å-step increments and 2-decimal formatting.

### `media/styles.css`
- **Step inputs** (`.bar-label .num-wrap`): width 38px, height 24px,
  `.num-steps` column 12px, font 12px. Matching `.bar-label` font/gap
  reduced for the tight floating-toolbar context. The inherited
  `.bar-input { width: 52px }` is overridden via more-specific selector.
- **Supercell** (`.sc-row` / `.sc`): switched from `grid 1fr 1fr 1fr`
  (stretch to panel width) to `flex` with each `.sc { width: 64px; flex:
  0 0 64px }`. Triplet now reads as a tight unit instead of spanning
  the panel.
- **`.prop-vis`** is visually hidden (`opacity: 0; pointer-events: none`) —
  the injected `.switch.sm` is the visible control. Same pattern as the
  `.toggle input` rule introduced in Feature 18.6b.
- **Bond `.num-wrap.bond-num`**: `flex: 1 1 0; height: 24px`, narrower
  step column (14px). Fits two side-by-side on a bond row alongside the
  `.switch.sm` + pair label.
- **Toolbar responsive**: `#top-bar` swaps `overflow-x: auto` → `flex-wrap:
  wrap; row-gap: 3px`. `container-type: inline-size; container-name:
  topbar` enables `@container topbar (max-width: 720px)` and
  `(max-width: 480px)` rules that progressively shrink button padding /
  height / font and hide the `Step(°):` label text (keeping the numeric
  stepper visible). When the side panel is open, `body.panel-open
  #top-bar` clamps `max-width` so the toolbar doesn't overlap the panel.

## Why this is a "polish iteration", not a v0.18.1 patch yet

All changes are tightenings of v0.18.0's surfaces, not fixes for
correctness regressions. Will roll into the next patch (v0.18.1) once
the user OKs the visual outcome and consolidates further feedback —
flagging here so the changes are captured in working/.
