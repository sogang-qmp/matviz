## 14.1 — 사이드 패널 레이아웃 모드 (offset vs overlay)

**파일:** `media/styles.css`, `src/editor/crystalEditorProvider.ts`, `src/webview/main.ts`, `src/webview/renderer.ts`.

**접근:**
1. offset 모드일 때 `<body>`에 `layout-offset` 클래스 부여. offset 모드에서 캔버스는 `left: var(--panel-width)`; overlay 모드에서는 기존처럼 캔버스 위에 패널이 떠 있음.
2. `--panel-width`는 `#side-panel` 리사이즈 이벤트에서 CSS 커스텀 프로퍼티로 노출.
3. 사이드 패널 헤더에 토글: "Layout: Overlay / Offset". 기본값 **offset**.
4. `vscode.getState()` / `setState()`로 저장 (기존 사용처 확인).
5. 레이아웃 모드 전환 시 `renderer.onResize()` 호출해 프레임버퍼를 새 CSS 크기에 맞춤.

**통과 기준:**
- 토글이 실시간으로 두 모드 전환.
- offset 모드에서 어떤 줌 레벨에서도 원자가 패널 뒤에 가려지지 않음.
- overlay 모드에서 패널 가장자리 드래그 리사이즈 기존대로 동작.
- 선택한 모드가 탭 닫았다 열어도 유지.

## 14.2 — 반응형 툴바 브레이크포인트

**파일:** `media/styles.css`, 필요 시 `src/editor/crystalEditorProvider.ts` (버튼 라벨).

**브레이크포인트 (에디터 폭):**
- **Compact** (<650 px): 아이콘만, 그룹 사이 구분자; 넘치는 그룹은 "…" 메뉴로 접힘.
- **Normal** (650–1200 px): 현재 동작 유지.
- **Wide** (≥1200 px): rotate-step / zoom-step 입력의 텍스트 라벨 완전 노출.

**접근:**
1. `#top-bar`에 CSS container query 사용 (VSCode 웹뷰 Chromium ≥ 105 지원 — 가능). 실패 시 media query로 폴백.
2. `.bar-btn--compact`, `.bar-btn--normal`, `.bar-btn--wide` 수식자 클래스 선언, 컨테이너 폭에 따라 CSS가 전환.
3. 오버플로우 메뉴는 "…" 버튼으로 여는 작은 드롭다운; 클릭 시 숨은 그룹 노출.

**통과 기준:**
- 에디터 폭 400 px에서 모든 버튼 접근 가능 (직접 또는 오버플로우 경유).
- 800 px에서 normal 레이아웃이 wrap 없이 들어맞음.
- 2000 px에서 라벨이 떠 있는 느낌 없음.

## 14.3 — 단축키 도움말 오버레이

**파일:** `src/editor/crystalEditorProvider.ts` (HTML + CSS), `src/webview/main.ts`.

**접근:**
1. `<div id="help-overlay" class="hidden">` 신규 추가 — 모달 위치: 배경 `rgba(0,0,0,0.6)` + 중앙 카드.
2. 카드 내용: **Navigation**, **Rotation**, **Zoom**, **Modes**, **기타**로 묶인 단축키 표. JS 상수 `SHORTCUTS`에서 채워 이벤트 리스너와 단일 소스로 유지.
3. 트리거: `?` 키 (Shift+`/`) 또는 모드바에 추가하는 `?` 버튼.

**통과 기준:**
- `?` 입력 후 100 ms 내 오버레이 표시.
- 현재 단축키 전부 나열.
- 두 팔레트 모두에서 가독성 유지.

## 14.4 — 상태 영속성 강화

**파일:** `src/webview/main.ts`, `src/webview/renderer.ts`, `src/editor/crystalEditorProvider.ts`.

**상태 스냅샷 스키마 (v1):**
```ts
interface PersistedState {
  schemaVersion: 1;
  style: string;
  camera: { mode: 'ortho' | 'persp'; target: [number, number, number]; position: [number, number, number]; up: [number, number, number] };
  palette: 'dark' | 'light';
  supercell: [number, number, number];
  visibility: { bonds: boolean; labels: boolean; polyhedra: boolean; boundary: boolean; cellDash: boolean };
  panel: { collapsed: boolean; width: number; layoutMode: 'overlay' | 'offset' };
  elementOverrides: { [symbol: string]: { color?: string; radius?: number; hidden?: boolean } };
  bondOverrides: { [pair: string]: { enabled: boolean; min?: number; max?: number } };
  iso: { level: number } | null;
  stepAngle: number;
  stepZoom: number;
}
```

**접근:**
1. `persistState(partial: Partial<PersistedState>)` 헬퍼 도입. `vscode.getState()`에 merge 후 `setState` 호출을 100 ms 디바운스.
2. `loadStructure` 시 상태 1회 읽어 입력/렌더러 채움.
3. 모든 변경 지점 (팔레트 토글, 슈퍼셀 변경, 오버라이드 편집, 카메라 이동 종료 등)에서 해당 슬라이스로 `persistState` 호출.
4. schemaVersion 체크 — 알 수 없는 버전은 폐기.

**통과 기준:**
- 탭 닫고 같은 파일 재오픈 → 모든 다이얼 복원.
- 파일 전환 시 파일별 상태는 초기화되지만 공용 선호 (팔레트, step 크기)는 유지 — 별도 키 사용: `matviz.global` / `matviz.file.<uri>`.

## 14.5 — 성능 예산 감사

**파일:** 수정 필요 없으면 로그만 `working/v0.14_feat5_perf-budget.md`.

**방법:**
1. 20 000원자 스트레스 픽스처 제작 (다이아몬드 원시셀 × 10×10×10 슈퍼셀).
2. `renderer.info.render` (draw call, 삼각형 수)를 `performance.mark` 하네스로 1회 캡처.
3. 기록: 회전 중 FPS, 정지 중 FPS, draw call 수, 삼각형 수, GC 후 힙 사용량.
4. 목표 미달 시 가장 큰 요인 1개 수정 후 재측정 — 과도한 최적화 금지.

**타깃** (Plan.md에서 수정 — `v0.14_ux-polish.md` 오픈 질문 4 참조):
- 20 000원자 원시 구조:
  - Draw call ≤ 200 (원안 100에서 완화; `InstancedMesh` 배치는 되지만 labels/sprite는 선형 증가)
  - 유휴 FPS = 0 프레임 (requestRender 게이트 존중)
  - GC 후 힙 ≤ 200 MB
- 1000원자 구조:
  - 회전 중 FPS ≥ 55 (SwiftShader 기준)
  - Draw call ≤ 50

## 14.6 — 5000원자 이상 결합 검출 UX

**파일:** `src/webview/renderer.ts`, `src/editor/crystalEditorProvider.ts` (Bonds 섹션 마크업), `src/webview/main.ts`.

**접근:**
1. 렌더러는 현재 원자 수 > 5000일 때 결합 검출을 즉시 스킵. 이 상태를 `getBondSkipInfo(): { skipped: boolean; atomCount: number; estimateMs?: number }` 게터로 노출.
2. Bonds 패널 섹션에 per-pair 리스트 위쪽 인라인 힌트 블록 추가:
   - 스킵 아닐 때 숨김.
   - 스킵일 때 `⚠ 결합 검출 스킵 — {N}원자가 5000 한계 초과. 예상 소요: {estimateMs} ms.` 표시.
   - 액션 버튼: **Compute anyway** → 강제 결합 빌드 실행.
3. 예상 시간은 100원자 부분 표본의 공간 해시 복잡도를 N² 계수로 외삽 (최대 10 s 상한).
4. "항상 계산" 선호는 14.4 스키마의 파일별 상태로 저장.

**통과 기준:**
- 6000원자 픽스처에서 힌트 노출; 버튼 클릭 시 결합이 예상 시간 ±50% 이내에 렌더.
- 한계 미만 구조는 오늘과 동일하게 동작.

## 14.7 — `AxisIndicator` 추출

**파일:** 신규 `src/webview/axisIndicator.ts`; `src/webview/renderer.ts` 축소.

**계약:**
```ts
class AxisIndicator {
  readonly group: THREE.Group;
  constructor(palette: ColorPalette);
  update(a: THREE.Vector3, b: THREE.Vector3, c: THREE.Vector3): void;
  setSize(px: number): void;
  setPalette(p: ColorPalette): void;
  dispose(): void;
}
```

**접근:**
1. `buildAxisArrows`, 축 화살표 material/geometry/texture, `updateAxisIndicator` 로직을 `renderer.ts` 밖으로 이전.
2. `AxisIndicator`가 자체 geometry/texture/material 리스트를 보유하고 재빌드 시마다 각각 `dispose()` 호출.
3. `renderer.ts`는 `AxisIndicator` 인스턴스 1개를 보유하고 업데이트를 위임.
4. renderer의 `materials`/`geometries`/`textures` 레지스트리에 축 자원이 더 이상 추적되지 않는지 확인 — 이들은 컴포넌트 내부로 이동.

**통과 기준:**
- `renderer.ts`가 60 LOC 이상 감소.
- 팔레트 토글 → 슈퍼셀 변경 → 팔레트 토글 루프를 50회 반복해도 `renderer.info.memory.geometries`, `.textures` 증가 없음.
- 시각적 회귀 없음.
