# v0.19 — Editor integration

## Goal

Make matviz behave like a first-class VSCode editor for crystallographers
who want to *edit* CIF / POSCAR / XSF text and see the 3D view follow:

1. **19.1 Split-pane**: text editor + 3D view side-by-side; edits to the
   text buffer reflow the 3D view (debounced reparse).
2. **19.2 Settings namespace**: `matviz.*` settings exposed via
   VSCode's settings UI; defaults configurable per-workspace.
3. **19.3 Undo/redo for property changes**: Ctrl+Z restores previous
   color / radius / cutoff edits made in the side panel. Separate from
   text-editor undo.

## Why now

Bundled with v0.20 (already implemented: spglib symmetry detection) and
shipped as a single **v0.20.0** tag per user reorder. v0.19's
editor-integration backlog has been the named successor to the v0.18
floating-UI work since v0.18 closed; this finally cashes it in.

## Roadmap impact

`Plan.md` v0.19 entry stays valid. After this lands, the deferred
marketplace publishing milestone (post-v0.21) becomes the next gating
event for any external visibility.

## Scope (3 features)

| # | Feature | Success criterion |
|---|---------|-------------------|
| 19.1 | Split-pane text + 3D | Edit a CIF in the text editor → 3D view re-renders within 300 ms (debounced). Save (`Ctrl+S`) writes the text buffer. Closing one pane closes the other (or both stay independent depending on architecture choice — see gate). |
| 19.2 | `matviz.*` settings | At least these defaults configurable: `matviz.defaults.style`, `.palette`, `.showBonds`, `.showBoundary`, `.bondCutoffPad`, `.isoLevel`, `.cameraMode`. Settings UI auto-renders from `package.json` `contributes.configuration`. localStorage persistence still wins for the current session; settings provide initial defaults on file open. |
| 19.3 | Property-panel undo/redo | Ctrl+Z restores previous element color / radius / per-pair bond min-max edits made in the side panel. Stack scoped to property edits — does NOT include camera changes, selection, supercell, display style. |

## Decision gate — 19.1 architecture (never self-approved)

Two viable paths; pick before any 19.1 code lands.

### Option A — `CustomTextEditorProvider` migration

Replace current `CustomReadonlyEditorProvider<CrystalDocument>` with
`CustomTextEditorProvider`. The webview and text buffer share the same
`TextDocument`; edits in either propagate via VSCode's standard change
events.

**Pros:**
- Single document — no synchronization questions about "which is canonical".
- Native VSCode dirty-state, save dialog, "Save All" all work.
- Future: `vscode.workspace.applyEdit()` from the side panel can push
  text-buffer edits (e.g., "change atom 5's color override stores a
  comment in the CIF source").

**Cons:**
- ~2-day migration touching document lifecycle, undo stack,
  custom-data flag handling. Higher risk per change.
- Read-only formats (`CHGCAR` binary, etc.) need separate treatment.

### Option B — Companion text editor

Keep the current `CustomReadonlyEditorProvider`. Open the matviz view
side-by-side with VSCode's regular text editor for the same URI. The
webview subscribes to `vscode.workspace.onDidChangeTextDocument` for
that URI and reparses + posts a structure update on every event.

**Pros:**
- Lower invasiveness — current architecture intact.
- Read-only and editable formats handled the same way.
- ~0.5-day delta.

**Cons:**
- Two views of the file (text + matviz) maintain their own state — if
  the user closes the text editor pane, the matviz view's reactivity
  breaks until reopened.
- Subtler edge cases: external file changes, file rename, multi-cursor
  edits during a parse — each needs explicit handling.

### Recommendation

**B (companion text editor)** — for this project specifically:
- Most matviz users *view* structures more often than they *edit*
  them; live edit is a "would be nice", not a primary workflow.
- The v0.21 Web SPA milestone will ship without VSCode entirely —
  CustomTextEditorProvider machinery doesn't carry over there. Keeping
  the editor adapter thin makes the SPA migration easier.
- 0.5-day budget vs. 2-day budget — the saved time funds 19.2 + 19.3
  comfortably.

The user makes the final call. The rest of this plan assumes
**B**; if **A** is chosen, the impl plan pivots to a TextDocument-
centric flow (the success criteria are unchanged).

## Out of scope

- **Live editing for binary formats** (CHGCAR, AECCAR, PARCHG): always
  read-only regardless of A/B choice.
- **Conflict resolution** when text edits change atom count mid-stream:
  best-effort reparse; if parse fails, the matviz view shows the last
  successful structure with a "(stale: parse error)" badge in the info
  pill.
- **Settings migration UI**: settings provide defaults; localStorage
  schema-v1 keeps wins on the same session. No automated migration
  banner.
- **Undo for camera / selection / supercell / display style** — the v0.19.3
  stack is property-edits-only.

## Tolerance defaults / constants

- 19.1 reparse debounce: 250 ms after the last keystroke.
- 19.2 setting keys (subject to bikeshed): `matviz.defaults.style`,
  `matviz.defaults.palette`, `matviz.defaults.showBonds`,
  `matviz.defaults.showBoundary`, `matviz.defaults.cameraMode`,
  `matviz.defaults.bondCutoffPad`, `matviz.defaults.isoLevel`,
  `matviz.defaults.axisIndicatorSize`.
- 19.3 undo stack depth: 50 entries.

## Exit criterion (combined v0.20.0 release)

1. Open a CIF in matviz, open the text editor side-by-side, edit
   `_cell_length_a` → 3D view reflows within 300 ms.
2. `matviz.defaults.style` set to `"space-filling"` in workspace
   settings → newly opened files default to space-filling.
3. Side-panel: tweak Si color → press `Ctrl+Z` → reverts. Re-press →
   re-applies (redo).
4. v0.20 spglib features all still work (cross-validation of 6 fixtures
   still passes — already verified).
5. CSP audit unchanged.
6. Clean type-check + build.

## Risks / open questions

- **19.1 race conditions**: rapid typing during a slow parse — debounce
  alone may not be enough. May need a `parseSeq` token to discard
  stale parse results.
- **19.2 setting/localStorage interaction**: when user opens a *file
  they've previously visited*, localStorage has saved state. If the
  user *also* changes a workspace setting, which wins? Plan: settings
  provide *initial* defaults; localStorage overrides after the first
  visit. Document this explicitly.
- **19.3 undo stack location**: pure-webview state vs. extension-host?
  Webview is simpler but doesn't survive editor reload. Plan: webview
  state, intentionally session-scoped (not persisted).

## Verification strategy

- **19.1**: manual edit/reflow timing on `nacl.cif`, `silicon.poscar`,
  `scf.in`. External rename test (rename file in Explorer while
  matviz open). Multi-cursor edit smoke test.
- **19.2**: change each setting in workspace settings, reopen a
  fixture, verify default applied.
- **19.3**: 20-cycle undo/redo stress on color/radius/bond-cutoff
  edits; verify state matches expected at each step.
