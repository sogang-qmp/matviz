# v0.20 — Symmetry detection (spglib WASM)

## Goal

Replace the `'P1'` fallback in `renderer.getStructureInfo()` with a
real spglib-driven detection so every loaded structure (POSCAR, XSF,
XYZ, PDB, Cube, CHGCAR, QE input/output, FHI-aims) reports its
correct space group, Hermann-Mauguin symbol, and Hall number in the
bottom-left info pill — not just CIF files that ship the symbol
inline.

## Why now

`'P1'` for everything-not-CIF is the most visible gap in the info
pill: a freshly-relaxed POSCAR or QE output reads as "no symmetry"
even when the structure is obviously cubic / hexagonal / etc. After
v0.18.x's polish pass closed all the chrome regressions, this is the
next-largest correctness issue and a prerequisite for the bonus
features that need symmetry context (Wyckoff labels, asymmetric-unit
extraction, primitive↔conventional toggle — all out of v0.20 scope but
enabled by it).

This is also the WASM artifact that v0.21 (Web SPA) will reuse — same
glue, same `.wasm`, deployed to GitHub Pages instead of the VSCode
webview. Doing v0.20 first means v0.21 inherits a working symmetry
backend rather than having to ship without one.

## Roadmap impact

Already documented in `Plan.md` as v0.20. Pulls ahead of v0.19
(editor integration) per the user's reordering decision (2026-04-27).
No prior version's exit criterion depended on v0.19 → v0.20, so the
swap is clean.

## Scope (5 features)

| # | Feature | Success criterion |
|---|---------|-------------------|
| 20.1 | Build/integrate spglib WASM artifact | `dist/spglib.wasm` (and glue JS) ship in the .vsix; the webview loads it on boot in <50 ms; bundle weight increase ≤300 KB compressed |
| 20.2 | `detectSymmetry(structure)` post-parser pass | Every parser that returns a `CrystalStructure` runs through `detectSymmetry`; result populates `spaceGroup`. CIF parser still defers to its own `_space_group_name_*` if present (CIF wins on conflict) |
| 20.3 | Info pill correct space group on non-CIF | Open `silicon.poscar` → pill shows `Fd-3m`. Open `nacl.cif` → still `Fm-3m` (CIF). Open `bcc-fe-fm.xsf` → `Im-3m`. The `'P1'` fallback in `getStructureInfo()` is removed |
| 20.4 | CLI renderer parity | `scripts/render.ts` loads the same WASM (Node FS API path); `--info` flag prints detected space group + Hall number to stdout. The headless renderer remains usable on CI machines without a browser |
| 20.5 | Architecture caveat docs (no CSP change needed) | spglib runs ONLY on the Node host (extension activation + CLI startup). The webview receives the pre-computed `spaceGroup` via `postMessage` and never loads WASM. Webview CSP is **unchanged** — this is the cleaner architecture compared to the original plan and removes one risk surface. CLAUDE.md updated to document the host-side architecture. |

## Out of scope (deliberately)

These are bonus surfaces that spglib makes possible but ship in a
later v0.20.x or v0.21+:

- **Wyckoff labels** in atom-select pill (`Si #5 (8a)`) — needs
  per-atom Wyckoff lookup beyond top-level symmetry.
- **Symmetry-refined cell** + standard-orientation auto-orient.
- **Asymmetric-unit extraction** for export (CIF/POSCAR write minimal
  set instead of all expanded atoms).
- **Point-group / Laue-class** display in info pill.
- **Conventional ↔ primitive cell** toggle in toolbar.
- **Tolerance UI**: a `matviz.symmetry.tolerance` setting will arrive
  with v0.19's settings namespace; v0.20.0 hard-codes 1e-3 Å (matches
  the existing CIF symmetry-expansion tolerance — `cifParser.ts`).

## Decision gates — RESOLVED 2026-04-27

**20.1 build path: D — `@spglib/moyo-wasm`** (locked).
Spglib org's official Rust port published as an actively-maintained
WASM npm package. Combines the strengths of options B (small bundle,
modern toolchain) and C (zero-toolchain integration) while neutralising
their respective downsides (third-party uncertainty, abandonment risk).
Wyckoff labels + symmetry operations exposed by the same `analyze_cell`
call, unblocking v0.20.x bonus surfaces at no extra cost.

**Bundle cap: 500 KB compressed** (C). Permissive headroom; expected
actual size ~150–250 KB so we have ample margin. Build fails the cap
guard if the .wasm exceeds it.

**Trajectory policy: A — frame 0 only with copy-to-all-frames**.
One detection per file load; replays don't recompute. Matches the
overwhelmingly common case (relax / MD around equilibrium where
symmetry is invariant per frame). Per-frame detection deferred to a
future v0.20.x toggle if a user asks.

**CLI `--info` output: 6-line text + `--info --json`** (default per
recommendation; revisit if the user's pipeline needs different fields).

**Fallback trigger** (revisit build path → option A emscripten):
1. measured WASM exceeds 350 KB compressed (well below the 500 KB cap
   but signals tooling regression),
2. cross-validation against `silicon.poscar`, `nacl.cif`, `bcc-fe-fm.xsf`,
   `bcc-cr-afm.xsf`, `scf.in`, `graphene.xsf` shows ≥1 mismatch with
   spglib upstream's known-good answers,
3. package becomes unpublished or ≥6 months stale.

## Tolerance defaults

- spglib position tolerance: 1e-3 Å in Cartesian space (matches
  matviz's CIF expansion tol).
- Angle tolerance: 1° (spglib default).
- Exposed as `matviz.symmetry.tolerance` once v0.19 introduces the
  settings namespace; hard-coded otherwise.

## Exit criterion

1. All `test/fixtures/*` files report a non-`P1` space group when the
   underlying structure is non-trivially symmetric (CIF, POSCAR
   silicon, AXSF Si trajectory, NaCl, the new bcc-fe-fm / bcc-cr-afm
   XSF fixtures, scf.in QE input). `'P1'` only appears for genuinely
   asymmetric structures (graphene defect cell, etc.).
2. The CLI renderer's `--info` output shows the same space group as
   the webview info pill for at least 5 fixtures (cross-validation).
3. A CSP audit (`grep "Content-Security-Policy"` in
   `crystalEditorProvider.ts`) confirms only `'wasm-unsafe-eval'`
   was added; `'unsafe-eval'` and `'unsafe-inline'` remain absent.
4. Bundle weight delta ≤300 KB compressed (`.vsix` size before vs.
   after).

## Risks / open questions

- **Bundle weight**: 300 KB is the soft target. Stretch is 200 KB if
  the chosen WASM build is well-optimized. If the chosen package is
  >500 KB, fall back to Option A and hand-tune the emcc flags.
- **WASM init in the CLI renderer**: Puppeteer's headless Chromium
  needs `--enable-features=WebAssembly` (already on by default) plus
  the existing `--allow-file-access-from-files` to read the .wasm via
  file:// URL. Should be straightforward but verify in the 20.4 spike.
- **CIF deference**: a CIF that ships an *incorrect* `_space_group_*`
  (corrupted file, hand-edited) would silently override spglib's
  correct answer. Acceptable trade-off; CIF authority is the
  documented behavior. If users complain, expose a "Recompute
  symmetry" command.

## Verification strategy

- **20.1**: bundle-size benchmark before/after; cold-start timing on
  the webview.
- **20.2**: unit-test `detectSymmetry` against 6+ canonical fixtures
  (silicon Fd-3m, NaCl Fm-3m, BCC Fe Im-3m, BCC Cr AFM, hex
  graphene P6/mmm, MAPbBr3 cubic Pm-3m).
- **20.3**: manual fixture sweep — open every `test/fixtures/*` and
  confirm space group in pill matches expectation.
- **20.4**: CLI cross-validation against the same 5 fixtures.
- **20.5**: `grep` audit + manual reproduction in a fresh VSCode
  install.
