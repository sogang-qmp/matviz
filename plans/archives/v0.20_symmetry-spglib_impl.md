## Architecture

Two execution contexts already established (CLAUDE.md):

- **Extension host** (Node.js, `dist/extension.js`): can load WASM
  via `WebAssembly.compile(fs.readFileSync(...))`. No CSP concerns.
- **Webview** (browser, `dist/webview.js`): loads WASM via
  `fetch('spglib.wasm')` then `WebAssembly.instantiateStreaming`.
  CSP `script-src` must allow `'wasm-unsafe-eval'` (the modern,
  scoped permission — *not* `'unsafe-eval'`).

The artifact ships in `media/spglib.wasm` (alongside other static
webview assets — wordmark, UI screenshot, etc.) plus a TS glue module
`src/shared/spglibWasm.ts` that exports a single `detectSymmetry()`
function consumed by both contexts. Same module, two callers.

- Modified: `.vscodeignore` — make sure `media/spglib.wasm` ships in
  the .vsix

### TS module shape (`src/shared/spglibWasm.ts`)

```ts
export interface SpglibSymmetry {
  spaceGroup: string;        // Hermann-Mauguin, e.g. "Fd-3m"
  spaceGroupNumber: number;  // 1..230
  hallNumber: number;        // 1..530
  pointGroup: string;        // e.g. "m-3m"
  crystalSystem: string;     // "cubic" | "hexagonal" | ... | "triclinic"
  origin: 1 | 2;             // setting choice for ambiguous groups
}

// Initialize the WASM module once per session. Idempotent.
export function initSpglib(wasmUrl: string): Promise<void>;

// Returns null if spglib could not detect symmetry (e.g. degenerate
// lattice). Caller should treat null as "fall through to P1".
export function detectSymmetry(
  lattice: [number, number, number][],
  positions: [number, number, number][],   // Cartesian Å
  species: string[],
  tolerance?: number,                       // default 1e-3 Å
): SpglibSymmetry | null;
```

### Loading recipe

- **Webview**: `initSpglib(webview.asWebviewUri(joinPath(extensionUri,
  'dist', 'spglib.wasm')).toString())`. Posted into the webview as part
  of the existing init message so `spglibWasm.ts` doesn't need to know
  the URL convention.
- **CLI renderer**: `initSpglib('file://' + path.resolve(__dirname,
  'spglib.wasm'))`. Puppeteer page receives the URL via the existing
  inline-page bootstrap.

### Bundle size guard

Add a build-time check in `esbuild.mjs`: if `media/spglib.wasm`
exceeds 350 KB raw, fail the build. This prevents accidental size
regressions when bumping the spglib version.

## Feature 20.2 — `detectSymmetry()` post-parser pass

### Files

- Modified: `src/parsers/index.ts` — wrap each parser's return value
  in a post-pass that calls `detectSymmetry()` and populates
  `spaceGroup` if the parser didn't already set it.
- Modified: `src/parsers/types.ts` — add `spaceGroupNumber?: number`
  and `pointGroup?: string` to `CrystalStructure` (info pill +
  bonus features will use them).

### Recipe

```ts
// In src/parsers/index.ts after each parseFoo() call:
function withSymmetry(structure: CrystalStructure): CrystalStructure {
  // CIF wins on conflict — its spaceGroup was set explicitly from
  // _space_group_name_*. Don't overwrite.
  if (structure.spaceGroup && structure.spaceGroup !== 'P1') {
    return structure;
  }
  const sym = detectSymmetry(
    structure.lattice,
    structure.positions,
    structure.species,
  );
  if (!sym) return structure;
  return {
    ...structure,
    spaceGroup: sym.spaceGroup,
    spaceGroupNumber: sym.spaceGroupNumber,
    pointGroup: sym.pointGroup,
  };
}
```

### Async consideration

`detectSymmetry()` is synchronous once WASM is initialized. The init
itself is async; the parser dispatcher must `await initSpglib(...)` at
webview boot before any parse-time symmetry call. Add an init gate in
`crystalEditorProvider.ts` that posts `{type:'spglibReady'}` to the
webview after init completes.

For the CLI renderer, `await initSpglib(...)` happens in the
`scripts/render.ts` startup (before `parseStructureFile`).

### Trajectory caveat

Multi-frame structures (`CrystalTrajectory`): run `detectSymmetry`
on frame 0 only; copy the result to all frames. Re-running per-frame
is wasteful and can produce flickering symmetry labels during
animation playback. Document this in the impl-doc (here) and the
inline comment.

## Feature 20.3 — Info pill correct space group on non-CIF

### Files

- Modified: `src/webview/renderer.ts` — `getStructureInfo()` returns
  `structure.spaceGroup ?? 'P1'` instead of always `'P1'` for non-CIF.
- Modified: `src/webview/main.ts` — info pill formatter consumes the
  same field; no markup change.

### Manual test cases

| Fixture | Expected space group |
|---|---|
| `silicon.poscar` | `Fd-3m` (#227) |
| `nacl.cif` | `Fm-3m` (#225) — CIF deferred, no spglib needed |
| `bcc-fe-fm.xsf` | `Im-3m` (#229) |
| `bcc-cr-afm.xsf` | `Pm-3m` (#221) — note: AFM moments are not in symmetry; non-magnetic spglib gives the parent symmetry |
| `scf.in` (MAPbBr3 ideal) | `Pm-3m` (#221) |
| `graphene.xsf` | `P6/mmm` (#191) — 2D periodicity flag may need handling |
| `mos2.xsf` | `P-6m2` (#187) |
| `buckyball.xyz` | `P1` (no periodicity → spglib returns null → fallback) |

### Edge cases

- **Single-atom unit cells**: spglib should return the lattice's max
  symmetry (e.g. `Im-3m` for monatomic BCC). Verify.
- **Non-periodic structures** (XYZ molecules, PDB without CRYST1):
  the lattice is the 20×20×20 fallback box. spglib will report `P1`
  trivially. Acceptable.
- **Slab structures** (XSF SLAB): only 2D periodicity. spglib's 3D
  detection should still produce a sensible answer with the existing
  out-of-plane lattice vector; note in commit message.

### Puppeteer WASM gotcha

Puppeteer's headless Chromium reads `file://` URLs only with
`--allow-file-access-from-files` (already in matviz's launch flags).
Verify the `.wasm` MIME type is served correctly; if Chromium balks,
fall back to inlining the WASM bytes as base64 in the bootstrap HTML.

## Feature 20.5 — CSP relaxation + caveat docs

### Files

- Modified: `src/editor/crystalEditorProvider.ts` line 202 (CSP meta
  tag) — append `'wasm-unsafe-eval'` to `script-src`.
- Modified: `CLAUDE.md` — add a bullet under "Critical caveats"
  documenting the relaxation and what it does NOT permit.

### Before / after

```diff
- "default-src 'none'; style-src ${webview.cspSource}; script-src 'nonce-${nonce}'; img-src ${webview.cspSource} blob: data:;"
+ "default-src 'none'; style-src ${webview.cspSource}; script-src 'nonce-${nonce}' 'wasm-unsafe-eval'; img-src ${webview.cspSource} blob: data:;"
```

### Caveat doc text (CLAUDE.md addition)

> **Critical caveat (v0.20+)**: webview CSP includes
> `'wasm-unsafe-eval'` for spglib. This permits `WebAssembly.compile`
> + `WebAssembly.instantiate` from same-origin sources. It does NOT
> permit `eval()`, dynamic `<script>` injection, or
> `new Function()` — those still require `'unsafe-eval'` which is
> deliberately absent.
